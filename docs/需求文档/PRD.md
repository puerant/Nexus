> 版本：v0.2 草稿
> 日期：2026-03-16

## 一、产品概述

### 1.1 产品定位

一款面向个人开发者/小团队的 **AI 驱动的项目管理工具**，将软件项目的完整生命周期结构化为五个阶段，并在每个阶段深度嵌入 AI 能力。核心理念是：用 AI 将项目前 30% 的方向和规划做扎实，再通过人机协作迭代完成剩余 70%。

### 1.2 核心用户

- **主要用户**：个人开发者
- **使用场景**：本地部署，管理多个并行开发项目
- **商业模式**：内部工具，非 SaaS

### 1.3 产品形态

- **Tauri 桌面应用**（跨平台原生桌面 App，macOS / Windows / Linux）
- 界面基于 WebView 渲染，用户体验与 Web 应用一致
- Rust 后端与数据库运行在本地，所有数据不离开本机

---

## 二、技术架构

| 层级 | 技术选型 | 说明 |
|---|---|---|
| 桌面框架 | Tauri 2.x | 跨平台桌面应用容器，WebView + Rust 核心 |
| 前端 | Vue 3 + TypeScript | 运行在 Tauri WebView 中，经典项目管理布局 |
| AI 调用 | Anthropic TypeScript SDK | 前端直接调用，处理对话流和流式输出 |
| 后端 | Rust（Tauri 核心） | 文件系统操作、Workspace 目录管理、系统权限管理 |
| 数据存储 | 本地文件系统（Workspace） | 所有产物以文件形式存储在用户打开的项目目录中，无需数据库 |
| AI 引擎 | 多模型可配置 | Claude 为主，支持切换 GPT-4 等 |

### 2.1 职责划分

**前端（Vue 3 + TypeScript）负责**：
- 所有 AI 对话调用（通过 Anthropic TS SDK 直接请求 Anthropic API）
- 流式输出接收与实时渲染
- 任务上下文的组装与管理

**Rust 后端负责**：
- 本地文件系统读写（写入被开发项目的代码文件及产物文档）
- Workspace 目录的打开、扫描与结构管理
- 系统级权限管理（限制文件访问范围仅在 Workspace 根目录内）

前端与 Rust 后端通过 Tauri 的 `invoke` 命令通信，无需 HTTP 服务器。

### 2.2 多模型支持策略

- 系统层面配置默认 AI 模型
- **每个阶段顶部均有独立的模型选择器**，可覆盖全局默认设置
- 模型选择粒度：系统默认 → 项目级覆盖 → 阶段级覆盖

## 四、五阶段核心流程

### 阶段一：需求梳理

**目标**：将模糊的业务诉求转化为结构化、可执行的需求文档。

**核心交互**：
- 内置 Chat 界面，模拟与客户的需求沟通场景
- AI 以 **主动追问模式**（AskUserQuestion）不断澄清需求中的模糊点
- 追问维度包括：功能边界、用户角色、优先级、技术约束、排除项等
- 多轮确认后，用户手动触发"锁定需求"操作

**输出产物**：
1. 结构化需求文档（Markdown 格式，保存至 Workspace 下的 `.ai-workflow/requirements/` 目录）
2. 功能思维导图（使用 **AntV G6** 渲染，AI 输出 Markdown 层级结构后自动转换为可视化图形）

**AI 工作流**：
```
用户输入初步想法
    ↓
AI 分析 → 识别模糊点 → 主动提问
    ↓
用户回答 → AI 更新需求草稿
    ↓（多轮迭代）
用户确认 → 锁定需求文档
```

---

### 阶段二：原型图设计

**目标**：根据锁定的需求，快速生成可交互的高保真原型。

**核心交互**：
- AI 读取需求文档，自动生成逐页面的可交互 HTML 原型
- 工具内嵌 iframe 预览，支持直接在浏览器中交互
- 用户可以描述修改意图，AI 更新原型并重新生成

**输出产物**：
- 一套可交互的 HTML 原型文件（每个主要页面/流程一个文件）
- **版本管理**：不自动保存版本，由用户手动打标签存档（如"v1-客户确认版"），仅存档版本可供回退

**约束**：
- 原型定位是沟通工具，不要求可直接转换为生产代码
- 优先覆盖核心用户流程，而非所有边缘情况

---

### 阶段三：技术选型

**目标**：结合需求特征与开发者技术栈，输出技术决策文档。

**核心交互**：
- AI 分析需求中的技术挑战点（并发、数据量、实时性、集成复杂度等）
- AI 结合用户在系统中维护的技术栈偏好，给出选型建议
- 每项选型包含：推荐方案、备选方案、选型理由、风险点

**输出产物**：
- 技术选型决策文档（Markdown）
- 包含：前端/后端/数据库/中间件/AI 接入方案/部署方案

---

### 阶段四：任务开发

**目标**：将需求拆解为可执行的最小开发任务，并由 AI 自动执行。

#### 4.1 任务拆解

```
需求文档
    ↓ AI 拆解
里程碑（Milestone）
    ↓ AI 拆解
模块（Module）
    ↓ AI 拆解
任务（Task）← 最小可执行单元
```

**任务结构**（每个 Task 包含）：
- 任务名称与描述
- 所属里程碑和模块
- 验收测试用例（AI 自动生成，人工可微调）
- 执行所用的提示词（来自预设模板 + 人工微调）
- 状态：待执行 / 执行中 / 待验收 / 已完成

#### 4.2 AI 自动执行模式（里程碑级）

- 人工设定里程碑目标后，触发"AI 自动执行"
- AI 按任务顺序依次执行：生成代码 → Rust 后端写入本地项目文件
- 执行过程以**类终端的流式输出**实时展示 AI 的思考和生成过程
- 遇到以下情况时，AI 暂停并等待人工确认：
  - 需要选择技术方案分支
  - 无法确定文件路径或模块归属
  - 任务依赖项尚未完成
  - 置信度低于阈值
- 人工确认后继续执行

#### 4.2.1 任务独立上下文窗口

**每个任务启动时，开启一个全新的、独立的 AI 上下文窗口**，确保任务之间互不干扰，每个任务都有干净的起始状态。

**上下文构建流程**：

```
任务开始执行
    ↓
① 从记忆模块检索相关内容（基于标签过滤）
    - 匹配维度：技术栈标签 + 业务领域标签 + 任务类型标签
    - 检索内容：提示词库中匹配的高效模板
                测试用例库中类似功能的验收用例参考
                经验摘要库中相关技术或业务的过往经验
    ↓
② 注入任务上下文
    - 当前任务描述与验收标准
    - 所属模块和里程碑目标
    - 项目技术选型文档摘要
    - 已完成任务的文件变更摘要（仅当前模块内）
    ↓
③ 组装 System Prompt 启动 AI 对话（Anthropic TS SDK）
    ↓
AI 执行任务 → 流式输出实时渲染到类终端界面
    ↓
生成代码 → invoke Rust 后端 → 写入本地项目文件
    ↓
AI 进行 Code Review
    ↓
任务完成 → 上下文窗口关闭 → 执行结果写入 Workspace 对应文件
```

**隔离的好处**：
- 避免长会话导致的上下文污染和 token 累积膨胀
- 每个任务可以独立重试，不影响其他任务
- 记忆模块提供的是精炼的结构化知识，而非原始对话历史

#### 4.3 Code Review

- 每个任务代码生成后，AI 自动进行初步 Code Review
- Review 维度：代码质量、安全漏洞、性能问题、是否符合验收用例
- Review 结果记录到任务详情中

#### 4.4 任务完成判定

- 任务对应的验收测试用例全部通过 → 任务标记为完成
- 测试用例可以是：单元测试代码 / 手动验收步骤清单 / E2E 测试脚本

#### 4.5 提示词管理

- 每个阶段/任务类型有预设提示词模板
- 用户可在执行前微调提示词
- 任务完成后，用户对本次提示词进行 **1-5 星人工打分**
- 评分 + 使用次数共同决定提示词在复用库中的推荐权重

---

### 阶段五：项目验收与复盘

**目标**：项目完成后沉淀经验，形成可复用的知识资产。

**核心功能**：

1. **自动聚合项目数据**
   - 所有任务执行记录
   - 所有使用的提示词及其效果评分
   - 所有测试用例
   - AI 对话历史
   - 遇到的阻塞点和解决方案

2. **AI 辅助生成复盘报告**
   - 里程碑完成情况统计
   - 技术决策回顾
   - 哪些 AI 提示词效果最好
   - 改进建议

3. **经验沉淀到四个复用库**

| 复用库 | 内容 | 使用场景 |
|---|---|---|
| 提示词库 | 高效提示词模板 | 新项目同类任务时推荐使用 |
| 测试用例库 | 历史验收用例 | 新项目类似功能时参考 |
| 任务模板库 | 成功的任务拆解方案 | 新项目快速生成任务清单 |
| AI 上下文库 | 项目经验摘要 | 作为新项目 AI 的背景上下文 |

---

## 五、知识积累飞轮

```
项目完成
    ↓
经验自动沉淀到四个复用库
    ↓
新项目启动时，AI 自动检索相关历史经验作为上下文
    ↓
AI 的规划更准确，代码生成质量更高
    ↓
项目完成更快、质量更好
    ↓（循环）
```

## 六、数据模型（文件结构）

### Workspace 目录约定

工具以打开本地文件夹（Workspace）的方式工作。所有 AI 产物和元数据均以文件形式存储，规范目录结构如下：

```
<workspace-root>/
├── .ai-workflow/               # 工具专用元数据目录
│   ├── project.json            # 项目配置（AI 模型、阶段状态、本地路径等）
│   ├── requirements/
│   │   └── requirement.md      # 结构化需求文档（含版本注释）
│   ├── prototypes/
│   │   ├── v1-初始版.html
│   │   └── v2-客户确认版.html  # 用户手动打标签的原型存档
│   ├── tech-design/
│   │   └── tech-stack.md       # 技术选型决策文档
│   ├── tasks/
│   │   ├── milestones.json     # 里程碑列表与状态
│   │   └── <task-id>/
│   │       ├── task.json       # 任务描述、状态、验收用例
│   │       ├── prompt.md       # 本次使用的提示词
│   │       └── review.md       # AI Code Review 结果
│   ├── memory/
│   │   ├── prompts.json        # 提示词库（含使用次数、评分）
│   │   ├── test-cases.json     # 测试用例库
│   │   ├── task-templates.json # 任务模板库
│   │   └── experiences.json    # AI 上下文经验摘要库
│   └── retrospective/
│       └── report.md           # 复盘报告
└── <用户项目代码文件...>        # 正常开发产物，AI 直接写入
```

### 核心实体（对应文件）

- **project.json**：项目配置，含五个阶段的状态、配置的 AI 模型、Workspace 根路径
- **requirement.md**：需求文档 Markdown，含锁定状态标记
- **prototypes/**：每个用户手动存档的原型为一个独立 HTML 文件
- **milestones.json**：里程碑列表，每条记录含名称、目标、关联任务 ID、状态
- **tasks/<task-id>/task.json**：任务描述、验收用例、执行状态、生成代码记录、Review 结果
- **prompts.json**：提示词模板，含关联阶段/任务类型、使用次数、效果评分
- **experiences.json**：经验摘要，含来源项目、摘要内容、技术栈/业务领域标签

## 七、非功能性需求

- **本地化优先**：所有项目数据以文件形式存储在用户打开的 Workspace 目录中，不依赖任何数据库或第三方服务
- **AI 调用透明**：每次 AI 调用的提示词、token 消耗、耗时都可以查看
- **渐进式使用**：每个阶段可以独立使用，不强制走完整流程
- **本地文件安全**：Rust 后端操作文件时，仅允许访问用户打开的 Workspace 根目录及其子目录
- **上下文隔离**：每个任务独立上下文窗口，防止长会话上下文污染，支持任意任务独立重试
